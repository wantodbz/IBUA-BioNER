# IBUA-BioNER

This repository contains the implementation of **IBUA-BioNER**, a
framework for few-shot Biomedical Named Entity Recognition (BioNER) that
combines Improved Balanced Undersampling (iBUS) and Adaptive
Mention-Replacement (ADAMER).

The repository includes the source code, datasets, grid search results,
and generated outputs used in the experiments.

## Repository Structure

``` text
IBUA-BioNER/

├── code/               # Jupyter notebooks for experiments
├── Dataset/            # BioNER datasets
```

## Folder Description

### `code/`

Contains all Jupyter notebooks used in the experiments.

-   `final-ibua-ncbi-full.ipynb` : Main experiment on the NCBI-Disease
    dataset.
-   `final-ibua-bc5cdr-part.ipynb` : Main experiment on the BC5CDR-Chemical
    dataset.
-   `final-ibua-bc2gm-part.ipynb` : Main experiment on the BC2GM
    dataset.
-   `GridSearch_K_NCBI.ipynb` : Grid search for the maximum augmentation
    budget **K**.
-   `GridSearch_RS_NCBI.ipynb` : Grid search for the iBUS sampling ratio
    **Rs**.

### `Dataset/`

Contains the datasets used in the experiments.

-   NCBI-Disease
-   BC5CDR-Chemical
-   BC2GM

### `gridsearch/`

Stores the CSV files containing the grid search results for **K** and **Rs**
reported in Section 5.1 and Fig. 4 of the paper.

### `outputs_NCBI/`, `outputs_BC5CDR/`, `outputs_BC2GM/`

Each output folder contains, for every few-shot setting (2%, 5%, 10%):

-   `split_indices_*.json` : Few-shot train/validation split indices used
    by all eight configurations.
-   `ibus_filter_log_*.csv` : Per-sentence logs from the iBUS filtering stage.
-   `adamer_aug_output_*.csv` : Augmented sentences generated by ADAMER.
-   `dataset_stats_*.csv` : Training-set composition for each of the eight
    configurations (sentence counts, entity/non-entity token counts,
    imbalance ratios, augmentation counts). These files underlie Tables 8-10.
-   `seed_details_*.csv` : Per-seed F1, precision, and recall for each of the
    eight configurations. These files underlie Tables 4-6 and the
    significance analysis in Tables 11-12.

## Reproducing the Reported Tables

-   **Tables 4-6** (main results): aggregated from `seed_details_*.csv`.
-   **Tables 8-10** (training-set composition): from `dataset_stats_*.csv`.
-   **Tables 11-12** (significance analysis): computed from the per-seed
    F1-scores in `seed_details_*.csv` using paired tests (paired t-test or
    Wilcoxon signed-rank, selected per comparison), with Benjamini-Hochberg
    correction.
-   **Fig. 4** (grid search): from the CSV files in `gridsearch/`.

## Implementation Notes

### Augmentation operator

`augment_sentence` builds each augmented copy left-to-right, copying the
non-entity spans between mentions verbatim and replacing (or keeping) each
mention in place. This guarantees that non-entity context tokens are never
overwritten, regardless of whether a replacement changes the mention length.

An earlier version of this function recomputed token offsets after each
replacement and, for sentences containing a multi-token replacement followed
by a further mention, could overwrite an adjacent non-entity token. The
`adamer_aug_output_*.csv` files were generated with that earlier version;
the failure-mode analysis in Section 5.6 of the paper quantifies the affected
fraction (below 0.6% of augmented tokens overall) and notes that, because the
operator is shared by the COSINER and ADAMER configurations, all
augmentation-based configurations were affected identically and the
comparison between them remains internally valid. The corrected operator in
this repository removes the defect for any future re-run.

### Reported metrics and prediction dumps

The reported metrics (Tables 4-6) are produced by `evaluate_model` on the
checkpoint with the best validation F1-score. No test-set prediction CSVs are
included: producing them requires a saved model checkpoint, which is not
retained under Kaggle "Save Version", so this optional path is disabled in
the notebooks.

## Environment

The experiments were run on **Kaggle Notebooks** with a single GPU, using
BioBERT v1.1 (`dmis-lab/biobert-v1.1`) as the encoder. The notebooks were
saved and executed through Kaggle's *Save & Run All (Commit)* feature
("Save Version").

Two consequences of this environment are relevant when reproducing the work:

-   **Dataset paths are Kaggle-specific.** The notebooks read the corpora
    from `/kaggle/input/...`. To run them elsewhere, attach the three corpora
    (NCBI-Disease, BC5CDR-Chemical, BC2GM) or edit the input paths at the top
    of each notebook to point to the files in `Dataset/`.
-   **Model checkpoints are not persisted.** Kaggle "Save Version" retains the
    committed outputs in `/kaggle/working/` but not intermediate model
    checkpoints, which is why the optional test-set prediction dump is disabled
    (see Implementation Notes above).

## Running the Experiments

Open the notebook corresponding to the desired dataset in the `code/`
directory and execute all cells. Each dataset notebook covers the three
few-shot settings through a single control variable near the top:

``` python
FEW_SHOT_FRACTION = 0.10   # set to 0.02, 0.05, or 0.10
```

Set this value and run the notebook once per setting. The five random seeds
(`SEEDS = [64, 128, 256, 512, 1024]`) are fixed, so the reported means and
standard deviations are reproduced exactly.
